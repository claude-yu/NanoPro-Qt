# 阶段 5 · 迁移决策（数据驱动）

_定稿：2026-06-01。依据：`docs/qt-benchmark.md`（实测）+ 本原型跑通的完整抠图闭环。_

## 一、要回答的问题
现 WebView 版（`../nanopro-editor`，SciEdit v1.4.0-beta，pywebview+WebView2+HTML/JS）够用，但用户质疑它「本质是网页、不像成熟软件」，性能与能力是否值得换原生栈（C++/Java/Rust/Python+Qt）。
**本阶段不盲目重写**，而是搭一个隔离原型（`qt-prototype/`，Python+PySide6+QGraphicsView+NumPy/OpenCV）实测关键性能路径，用数据决定。

## 二、决策门槛（阶段0 预先定好，事后不得改）
1. 关键场景提升 **<30% → 不全量迁移**。
2. 若**只**算法提升 → 优先「WebView UI + OpenCV 算法层」混合，不动 UI。
3. **4K/多图层交互显著提升 且 复杂度可控 → 渐进迁移 PySide6**。
4. 无实测数据不得宣布重写。

## 三、实测结果（对照门槛）
| 维度 | 门槛 | 实测（用户本机 Win11 / Py3.13 / PySide6 6.11） | 结论 |
|---|---|---|---|
| 渲染/交互 FPS | 4K+20 图层 ≈50 FPS | **70 图层连续动画 173–188 FPS**、导入 18 ms | 约 **3.5×** 超标，决定性胜 |
| 魔棒选区 | 2K<300ms / 4K<1000ms | 2K **20 ms** / 4K **81 ms** | 碾压 |
| 去背景/抠出 | （提取）合理即可 | 4K alpha 提取 **63 ms**、羽化 **1.3 ms** | 碾压 |
| 导出 PNG | 4K<3000ms | 4K 编码 **70 ms**（多层合成另计，仍 <数百 ms） | 碾压 |
| 图层上画笔/橡皮 | WebView 难做 | Qt 天然支持（在层 QImage 上绘制，alpha 正确） | 新增能力 |
| 完整抠图闭环 | 须跑通 | 导入→图层→画笔→**魔棒选区→去背景（透明背景图，棋盘格可见）/抠出新层**→合成导出，已实跑成功 | ✅ |

补充工程结论（实跑暴露并已解决）：
- **大图涂抹流畅的关键**：图层用 `QGraphicsItem.paint(drawImage)` + 画笔只 `update(脏矩形)`，**不要**每段 `QPixmap.fromImage()` 整图重传（后者在 1448×1086 上明显卡）。
- **透明可见性**：画布需棋盘格背景，否则去背景效果在深色底上看不出（曾被误判为「魔棒无用」）。
- **图层可管理性**：必须有激活层可视标识（青虚线框）+ 删除/选择，否则多层错位时用户无法理解魔棒「作用在哪层」。

## 四、决策
命中门槛 **3**（渲染 3.5× 且算法全面提升，二者皆显著；原型复杂度可控——核心仅 `main/editor_window/canvas_view/layer_item/image_ops` 五个文件即跑通闭环）。
→ **采用「渐进迁移到 PySide6」**，而非门槛 2 的纯混合（因为不仅算法快，渲染/交互这一 WebView 最难优化的部分也决定性胜出）。

**不是推倒重来**：WebView v1.4.0-beta 作为**当前可发布产品**继续保留，迁移期并行存在，能力对齐后再切换默认。

## 五、为什么是 PySide6 而非 C++/Rust/Java
- **C++/Qt**：性能上限最高，但开发/迭代成本数倍，且本场景瓶颈在「渲染框架 + 像素算法」——PySide6 的 QGraphicsView 与 C++ 同引擎，OpenCV 本就是 C/SIMD，**热路径已是原生速度**，Python 只做编排，差距可忽略。
- **Rust**：生态（GUI、图像、PSD/AI 文件解析）远不如 Qt+OpenCV 成熟，绑定与迭代成本高；真有 CPU 热点可单点 Rust→pyd 提速，无需整体换栈。
- **Java**：桌面图像生态弱于 Qt，无优势。
- **结论**：PySide6 = 原生级性能 + 最低迁移摩擦 + 复用现有 Python/OpenCV/AI 代码。必要时局部上 Rust/C++ 扩展。

## 六、迁移工作量预估（原型→产品需补的）
已验证（可直接长成产品）：原生窗口、缩放/平移、图层模型、画笔/橡皮、魔棒/去背景/抠出、棋盘格、合成导出、图层删除/激活标识、多图层性能。
**待建**（粗估，单人）：
- 撤销/重做（命令栈）——★高优先，约 1 周。
- 文字图层 / 矢量标注（箭头、框、比例尺）——2–3 周。
- 完整图层面板（缩略图、显隐、不透明度、排序拖拽、分组、混合模式）——1–2 周。
- AI 生成接入（复用现有 grsai 调用 + Key 本机存储逻辑，套进 Qt 面板 + QThreadPool 后台）——1 周。
- OpenCV 重运算移入 QThreadPool 后台线程（不卡 UI）——3–5 天。
- 打包（PyInstaller/Nuitka）、启动时间、跨平台验证——1 周。
- 完整魔棒（加选/减选/全局选同色/蚂蚁线）——1 周。

## 七、风险与未验证项（迁移前须正视）
- 本原型**未**验证：撤销系统的内存占用（大图多步）、超大画布（>8K）滚动、文字渲染质量、打包体积/启动速度、跨平台（macOS/Linux）一致性。
- 超大图纯缩放时 `ImageLayerItem` 每帧 `drawImage` 无 pixmap 缓存——必要时加 `cacheMode`（已记入 progress 遗留项）。
- AI 生成是当前产品核心卖点，迁移期必须保证其在两版间不退化。

## 八、下一步（建议）
1. 锁定本决策；WebView v1.4.0-beta 维持为发布版。
2. 在 `qt-prototype` 基础上起正式工程骨架，**第一件事做撤销/重做**（无撤销不算编辑器）。
3. 接 AI 生成 + 后台线程化 OpenCV，达到「能替代 WebView 日常使用」后再切默认。
4. 复杂功能（PSD 兼容、矢量、批处理）按需排期。
